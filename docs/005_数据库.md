# 数据库设计指南 (Industrial Grade V2.2)

## 数据库概述

LOF 基金套利监控系统采用 **"大一统超级宽表"** 与 **"模块化职责分离"** 的数据库设计哲学。系统默认使用 SQLite 数据库，并强制开启 **WAL (Write-Ahead Logging)** 模式，以支撑多线程高频写入与查询。

**数据库路径**：`arbcore/database/arb_master.db`

---

## 模块化管理器 (DatabaseManager)

为了提升代码的可维护性，数据库接口不再堆叠在单一类中，而是通过 `组合模式` 拆分为三个专业子管理器：

### 1. 基金管理器 (`db.funds`)
- **职能**：管理 A 股基金的价格、净值、估值因子及底层篮子权重。
- **核心表**：`fund_data`, `fund_daily_factors`, `fund_basket_weights`

### 2. 市场管理器 (`db.market`)
- **职能**：管理宏观汇率、海外 ETF 收盘价、指数 K 线及期货结算价。
- **核心表**：`exchange_rate`, `usa_etf_daily_prices`, `futures_daily`, `index_daily`

### 3. 系统管理器 (`db.system`)
- **职能**：维护 API 访问状态、系统健康日志及原始数据备份（Data Lake）。
- **核心表**：`access_sync_status`, `system_health`, `raw_api_data`

## 核心数据字典

### 1. fund_data (基金主表)
*系统中最重要的超级宽表，每一行代表一只基金在一个交易日的完整成绩单。*

| 字段名 | 类型 | 说明 |
|--------|------|------|
| date | TEXT | 日期 (YYYY-MM-DD，联合主键) |
| fund_code | TEXT | 基金代码 (联合主键) |
| price | REAL | 盘中现价 / 盘后收盘价 |
| nav | REAL | 官方日终净值 |
| static_val | REAL | 静态官方估值 (根据算法推演) |
| val_error | REAL | 估值误差 (官方净值 vs 静态估值) |
| premium | REAL | 盘后溢价率 ( (price - nav)/nav ) |

### 2. fund_daily_factors (估值因子表)
*Woody API 提取后的常量池，是前端秒开渲染的基石。*

| 字段名 | 类型 | 说明 |
|--------|------|------|
| fund_code | TEXT | 基金代码 |
| calibration | REAL | 校准值 (Calibration) |
| hedge | REAL | 对冲比例 (Hedge) |
| position | REAL | 基础仓位 (Position) |

### 3. usa_etf_daily_prices (海外 ETF 价格表)
*记录美股底层标的的基准价格，特别支持“虚拟后缀”替身机制。*

| 字段名 | 类型 | 说明 |
|--------|------|------|
| symbol | TEXT | 标的代码 (如 GLD, ^GLD-EU) |
| price | REAL | 收盘价 (或特定时点快照价) |
| netvalue | REAL | 官方净值 (若有) |

## 性能优化与设计哲学

1. **动静结合 (Upsert Strategy)**：
   - 盘中使用 `INSERT OR REPLACE` 实时覆写当天行，保持数据新鲜。
   - 盘后通过 `UPDATE` 精修官方字段（如 `nav`），实现无缝过渡。
2. **快慢分离**：
   - 指数行情分为“实时快照表”（单行缓存）与“历史 K 线表”（分段归档）。
3. **安全隔离**：
   - ETF 轮动模块使用物理隔离的 `etf_raw_api_data` 表，确保策略私密性。
4. **全自动清理**：
   - `db.cleanup_old_data(days=30)` 自动回收陈旧的常规行情数据，防止数据库膨胀。
